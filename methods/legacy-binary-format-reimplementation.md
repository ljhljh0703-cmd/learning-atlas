# 레거시 바이너리 포맷 재구현 방법론

> 구 게임 포맷·독점 컨테이너(MPQ 등) 같은 **레거시 바이너리를 재구현/재현**할 때 반복 재사용되는 원칙. uzmap-forge(WC3 `.w3x`) 실측에서 일반화. 특정 포맷 지식이 아니라 *접근 태도*가 자산.

## 1. "안 열림 ≠ 보호맵" — 리더 견고성을 먼저 의심하라

naive 파서가 파일을 못 열면 "보호/암호화됐다"로 성급히 결론내기 쉽다. **실제 원인은 리더 버그일 때가 많다.**

- **실증**(uzmap-forge R-102→R-102b): 서브에이전트가 GemTower 맵을 "보호맵·표준 리더로 열람 불가"로 보고 → 작가 "비보호 맵" 지적 → 독립 재조사 결과 **맵은 완전 정상, 리더 버그 2종이 원인**.
  - 버그① 오염된 장식 헤더필드(headerSize/formatVersion)로 진짜 헤더를 기각 → 검증을 *필드 일치*가 아니라 *해시테이블 복호화 매치*로 교체.
  - 버그② `blockTablePos = 0xfffff620` = 32비트 랩어라운드(테이블이 헤더 앞 절대오프셋) → 오프셋 계산 전부 `>>>0`.
- 흔한 프로텍터는 정상 재생되는 맵을 *naive 파서만* 튕긴다(오염 장식필드 + 음수 랩어라운드 오프셋). 표준 로더(원본 게임·StormLib)가 이미 하는 견고 오픈을 구현하면 대개 열린다.
- **교훈**: 실패를 "대상이 나쁨"으로 귀속하기 전에 *내 리더가 규격만큼 견고한가*를 먼저 검증. → [검증자의 주장도 환각이다 — 강한 주장은 1차 출처로 재-Gate](verifier-claims-need-regate.md)(서브에이전트 결론 불신·독립 재현)의 도메인 인스턴스.

## 2. clean-room 윤리 경계선 — 규격 준수 vs 적극 deprotect

레거시 포맷 재구현의 합법·윤리 경계는 명확히 선을 그을 수 있다.

- **허용(규격 준수)**: 표준 리더가 *이미 하는* 견고 오픈 — 오염필드 무시·정렬 스캔·32비트 랩어라운드·해시테이블 복호화 검증. 정상 데이터를 정상적으로 읽는 것.
- **금지(적극 deprotect)**: listfile 복원으로 숨김파일 열거·미지 키 브루트포스·트리거 역난독화·재배포용 보호 제거. 저작자가 *의도적으로 숨긴 것*을 파헤치는 행위.
- **집행은 사람이 아니라 게이트**: 원저작물 파생 바이너리 커밋을 말로 금지하지 말고 glob 검사 + `.gitignore` 이중으로 기계 차단. 라이선스 경계는 게이트가 지킨다.

## 3. 수요주도 커버리지 — API 롱테일은 "실제 호출"만

레거시 API 재구현 시 명세 전체(예: natives ~2,000종)를 다 구현하려 들지 말고 **대상이 실제 호출하는 것만** 구현한다.

- 미구현 호출을 조용히 무시하지 말고 `UnimplementedNative` 같은 **이벤트로 집계** → 커버리지 리포트 → 다음 작업 백로그로 순환.
- 스코프를 수요가 정의하게 하면 롱테일이 잘리고 진행 가시성이 생긴다.

## 4. 의존성 제로 게이트 = 이식성

- 샌드박스에서 패키지 레지스트리가 막히면(403 등) 언어 내장 기능만으로 전환(node 내장 test·strip-types·zlib 등).
- 결과: 게이트가 환경 무관(샌드박스=작업자 로컬=CI)으로 재현 가능. 오프라인/폐쇄망 재구현 전반에 재사용.

## 5. orientation은 힌트, 정본은 실바이트

- 세션 오너/디스패치가 준 기술 orientation(예: "이 필드를 신뢰")이 실맵 바이트와 충돌하면 → orientation을 *검증 대상*으로 취급하고 정본 소스(StormLib 원문 등)를 재확인해 정정.
- 디스패치 프롬프트의 "verify against real bytes; do not trust blindly" 문구가 이 동작을 유도 — 기술 디스패치 표준 슬롯으로 재사용 가치.
